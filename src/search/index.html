Demo Details

Search forms can be one of the most important parts of a UI, as a good search interface can allow users to quickly find the content they are looking for without the need to parse through content or navigation links in hopes of finding what they need.

To help promote the existence of a search form, the search landmark (role="search") should be applied to the wrapping form element of the search UI. Beyond that, the most basic of search forms might look like the following markup: 

<form role="search">
  <input type="search"
    id="srch"
    name="srch"
    aria-label="Search for...">
  <button type="submit"
    id="srch_btn"
    aria-label="Submit Search">
    <svg viewBox="50 461 23 23"
      aria-hidden="true"
      focusable="false">
      <path d="..."
        id="icon_search"
        stroke="none"
        fill-rule="evenodd"></path>
    </svg>
  </button>
</form>

In many search forms there may be no explicit label, as the presence of the search icon (within the submit button) gives the visual indicator to users that this is a search component. However, an aria-label should still be applied to the input[type="search"] to give the input an accessible name.

An aria-label is also used on the button[type="submit"] to give it an accessible name as well. The SVG within is hidden from screen readers and given an focusable="false" attribute to help ensure that Internet Explorer won't allow keyboard focus to enter into the SVG itself.

Note that the input[type="search"] and role="search" are not the same thing. As mentioned, the role="search" surfaces the form as a landmark, and should not be applied to the input, as that would negate the input as being announced as an editable 'text field'.

More complex search forms might contain additional form controls, or might utilize a combobox in place of, or in addition to, a standard input[type="search"] form control. Search forms should have a submit button as a consistent indicator in how a form can be submitted.
